# DX-03 — Resumo Telegram e entrega automática em branch após aprovação

Status: implementada, com limitações operacionais documentadas abaixo.

## Resultado

O gate Telegram recebe resumo técnico fragmentado e sanitizado, com arquivos,
estatísticas, testes, reviewer, findings, riscos e documentação. Projetos podem
exigir documentos no snapshot e habilitar explicitamente:

```toml
[delivery]
mode = "push_branch"
remote = "origin"
base_branch = "main"
branch_template = "{task_slug}"
commit_message_template = "{task_id}: {task_title}"
push_after_human_approval = true
```

Após decisão válida, o delivery revalida decisão, `HEAD`, hash e manifesto,
constrói uma index temporária a partir do base commit, grava um único commit,
cria a branch local e usa refspec explícito sem force. `main`, `master` e a base
configurada não podem ser alvo. O OID remoto é confirmado antes de `PUSHED`.

Projetos sem `[delivery]` continuam terminando em `HUMAN_APPROVED`, sem commit,
branch ou push.

## Retomada

Falhas ficam em `DELIVERY_FAILED` sem perder a aprovação. Se `commit_oid` já foi
registrado, uma retomada o reutiliza após validar tree e parent; falhas anteriores
à criação do commit naturalmente não preservam commit local. `resume` executa
somente delivery e nunca repete Cursor ou Codex nesse estado.

## Limitações operacionais atuais

- a unidade systemd libera escrita somente no state root, mas delivery precisa
  escrever objetos e refs no repositório; uma aprovação processada somente pela
  unidade pode falhar e deve ser retomada fora dela;
- o token Telegram carregado por `EnvironmentFile` está no ambiente da ponte, e
  hooks Git locais podem herdá-lo durante delivery;
- Git remoto não possui timeout nem modo não interativo explícito;
- envio Telegram é *at-least-once*: há uma janela de duplicação entre enviar e
  persistir `message_id`, o offset não sobrevive a restart e não há singleton;
- o botão mantém o texto “Aprovar e publicar branch” mesmo quando delivery está
  desabilitado; nesse caso ele apenas registra `HUMAN_APPROVED`.

A correção preferida é mover delivery para um worker separado, sem credencial
Telegram, com ambiente Git mínimo, hooks desabilitados, timeout e acesso de
escrita somente ao repositório selecionado.

## Verificação
