# DX-06 — Worker Git endurecido e unidades systemd separadas

Status: planejada.

Marco: M0, parte 2 de 2.
Depende de: [DX-05](DX-05.md).
Próxima task: [DX-07](DX-07.md).

## Contexto

A DX-05 remove Git do callback, mas o worker ainda precisa de uma fronteira
operacional segura. Ele deve escrever no repositório selecionado, usar a
autenticação Git existente e nunca receber o token Telegram ou executar hooks
locais.

Operações remotas também precisam terminar de forma previsível quando há prompt
interativo, DNS lento, SSH indisponível ou remote travado.

## Objetivo

Executar delivery em processo e unidade próprios, com:

- ambiente mínimo e sem credenciais Telegram;
- hooks Git desabilitados;
- comandos não interativos e com timeout;
- acesso de escrita restrito ao project state e Git common dir;
- teste end-to-end dentro do sandbox systemd real.

Ao final desta task, o marco M0 estará concluído.

## Escopo obrigatório

### 1. Executor Git seguro

Centralizar todas as chamadas Git de delivery em uma função que:

- constrói ambiente a partir de allowlist explícita;
- nunca herda `AGENT_TELEGRAM_*`, env de projeto ou `GIT_*` arbitrárias;
- permite somente variáveis operacionais necessárias, como locale, `HOME`,
  `PATH`, `SSH_AUTH_SOCK` e XDG aprovado;
- define `GIT_TERMINAL_PROMPT=0` e modo não interativo do credential manager;
- desabilita hooks com `-c core.hooksPath=/dev/null`;
- recebe timeout obrigatório por operação;
- encerra o grupo de processos Git no timeout;
- captura stderr, sanitiza e retorna motivo estruturado sem credenciais.

Aplicar a função a `rev-parse`, `hash-object`, `update-index`, `commit-tree`,
`update-ref`, `ls-remote` e `push`. Não podem restar chamadas Git de delivery
fora desse executor.

### 2. Política de timeout

Adicionar configuração segura com defaults e limites para:

- operações Git locais;
- descoberta do remote;
- push;
- encerramento gracioso.

Timeout deve resultar em `DELIVERY_FAILED` com motivo específico, preservar
decisão/job/worktree e permitir novo `resume`. Prompt interativo não pode manter
o worker bloqueado.

### 3. Unidade da bridge

Manter a bridge global com:

- credential file Telegram;
- escrita somente no state root global;
- nenhum acesso de escrita aos repositórios;
- nenhum import/comando de delivery;
- singleton de unidade por state root documentado.

### 4. Unidade do worker por projeto

Adicionar geração explícita, sem instalar ou habilitar automaticamente:

```bash
agent-loop systemd-worker-unit \
  --repo /projetos/alvo \
  --state-root ~/.local/state/codex-cursor-agent-loop \
  --output ~/.config/systemd/user/agent-delivery-<repo-id>.service
```

A unidade deve:

- não carregar o credential file Telegram;
- apontar para um único project state/repositório canônicos;
- permitir escrita somente no project state e Git common dir necessários;
- manter home e sistema read-only fora dessas exceções;
- usar `NoNewPrivileges`, `PrivateTmp` e hardening compatível;
- reiniciar somente em falha operacional recuperável;
- recusar paths com escaping inválido, symlink ambíguo ou marcador não resolvido.

Documentar geração, revisão manual, enable/start, logs e remoção.

### 5. Autenticação Git

Usar somente autenticação já configurada pelo operador. Não copiar chave, token,
credential helper output ou URL de remote para o run.

O preflight/`doctor` mínimo desta task deve diagnosticar:

- remote e push URL;
- autenticação não interativa disponível ou ausente;
- Git common dir e permissões;
- hooks/configuração perigosa que será ignorada;
- token Telegram ausente do ambiente efetivo do worker.

Diagnósticos devem mostrar nomes/estado, nunca valores.

### 6. Teste systemd real

Adicionar teste de integração Linux que:

1. cria repo e remote bare temporários;
2. inicia bridge e worker em unidades/scope com hardening real;
3. simula callback autorizado;
4. confirma resposta rápida e job pendente;
5. confirma branch remota e OID;
6. confirma `main` inalterada;
7. confirma que hook de teste não foi executado;
8. confirma que o worker não recebeu token Telegram.

Se systemd não estiver disponível no ambiente de desenvolvimento, o teste pode
ser marcado por requisito explícito, mas deve ser obrigatório no CI Linux antes
de concluir M0.

## Invariantes de segurança

- processo/unidade com token Telegram nunca executa Git;
- worker não possui token Telegram, env de testes ou segredos do executor;
- hooks locais e globais não executam durante delivery;
- nenhum comando Git aguarda prompt;
- timeout nunca promove `PUSHED`;
- `ReadWritePaths` não pode liberar home, `/`, todos os projetos ou globs amplos;
- remote/branch continuam congelados e revalidados;
- push permanece explícito, sem force, tag, merge ou base branch.

## Compatibilidade e rollback

- foreground worker continua suportado quando systemd não é usado;
- falha ao gerar/usar unidade não altera run nem repositório;
- remover a unidade preserva state, jobs, worktrees e branches;
- runs pendentes podem ser concluídos pelo worker foreground;
- templates antigos da bridge recebem migration/documentação clara.

## Fora de escopo

- cgroups para Cursor/Codex e limites gerais de recursos, tratados na DX-09;
- singleton/outbox avançado do M3;
- armazenamento de credenciais Git;
- suporte oficial a sistemas sem isolamento equivalente;
- merge, PR ou deploy.

## Critérios de aceite

1. Bridge e worker são processos/unidades distintos.
2. Worker não recebe token Telegram nem variáveis de projeto.
3. Nenhum hook Git executa durante delivery.
4. Remote travado ou prompt termina em timeout recuperável.
5. Unidade do worker conclui delivery com permissões mínimas.
6. Unidade da bridge continua sem escrita no repositório.
7. Push cria apenas a branch/commit aprovados e confirma OID.
8. Teste systemd end-to-end é obrigatório no gate M0.

## Testes obrigatórios

- inspeção do ambiente efetivo de cada unidade;
- `pre-push`, `reference-transaction` e `core.hooksPath` configurado;
- helper/SSH com prompt interativo;
- timeout de `ls-remote` e push;
- stderr contendo URL/token fake e redaction;
- repo/common dir com espaços e caracteres systemd relevantes;
- remote bare, branch idêntica/diferente e concorrência;
- bridge sem permissão de escrita no repo;
- worker sem permissão fora do repo/project state;
- `systemd-analyze verify`, `bash -n`, `git diff --check` e suíte completa.

## Entrega obrigatória

- executor Git central e política de ambiente/timeout;
- template(s) e comandos de geração da unidade;
- procedimento de instalação e rollback;
- teste end-to-end real;
- atualização do threat model e documentação operacional;
- evidência de ausência do token e de hooks não executados.
