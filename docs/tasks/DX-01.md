# DX-01 — Notificações e gate humano via Telegram

Status: implementada; documento histórico do escopo original.

As exclusões de commit e push abaixo valem para a entrega DX-01. O
[`DX-03`](DX-03.md) acrescentou posteriormente, de forma opt-in, commit exato e
push somente para uma branch da task após aprovação humana. Merge, push na base,
force-push, PR e deploy continuam proibidos.

## Contexto

O loop local `scripts/agents/run_task.sh` executa Cursor e Codex em worktree
isolado, mas hoje exige acompanhamento pelo terminal. O operador precisa receber
o resultado no celular e confirmar o encerramento humano da task sem ganhar um
terminal remoto ou uma interface capaz de executar comandos arbitrários.

Esta é uma melhoria de developer experience. Ela não pertence ao produto SaaS
e não pode introduzir dependência do Telegram no runtime da plataforma.

## Objetivo

Adicionar uma ponte local de Telegram que:

1. notifique mudanças terminais do loop de agentes;
2. apresente um botão de aprovação quando a revisão técnica terminar;
3. registre uma decisão humana autenticada e auditável;
4. permita que o loop encerre como humanamente aprovado sem executar a próxima
   task, integrar Git ou ampliar poderes do bot.

## Fluxo esperado

```text
Cursor executa
  -> Codex revisa
  -> APPROVED técnico
  -> AWAITING_HUMAN_APPROVAL
  -> Telegram envia resumo + botão
  -> operador autorizado aprova
  -> HUMAN_APPROVED
  -> loop encerra e preserva o worktree para o planner integrar
```

Em `BLOCKED`, falha de executor/revisor ou limite de iterações, o bot envia
somente uma notificação com o caminho do relatório. Esses estados não oferecem
botão de aprovação.

## Escopo obrigatório

### 1. Contrato de aprovação

- Distinguir aprovação técnica de aprovação humana.
- Após `APPROVED` do Codex, gravar no run directory uma solicitação contendo,
  no mínimo: task, run id, base commit, worktree, relatório da revisão e hash do
  diff aprovado.
- Usar escrita atômica para solicitação e decisão.
- Vincular a decisão ao run id e ao hash do diff; aprovação antiga ou replay
  não pode liberar outro diff.
- Tornar callbacks idempotentes.
- O estado final humano deve ser inequívoco (`HUMAN_APPROVED`).

### 2. Ponte Telegram

- Usar Bot API por long polling; nenhuma porta HTTP pública ou webhook.
- Aceitar apenas chat privado e `user_id`/`chat_id` numéricos configurados.
- Ignorar username como identidade de segurança.
- Enviar notificações para `AWAITING_HUMAN_APPROVAL`, `BLOCKED` e falhas.
- Em aprovação pendente, mostrar botão inline com identificação opaca,
  limitada e de uso único.
- Responder de forma neutra a remetentes não autorizados sem revelar caminhos,
  tasks, logs ou estado do host.
- Truncar mensagens e nunca enviar ambiente, credenciais ou logs completos.

### 3. Segredos e processo

- Ler token e allowlist somente de ambiente ou credential file externo ao Git.
- Não incluir token real em exemplos, testes, logs ou relatórios.
- Documentar execução como usuário, nunca root.
- Fornecer unidade `systemd --user` ou procedimento equivalente, sem instalar ou
  habilitar serviços automaticamente.
- Falha do Telegram não pode corromper o run directory nem converter uma task em
  aprovada.
- O loop deve ter timeout configurável para aguardar decisão humana e preservar
  `AWAITING_HUMAN_APPROVAL` quando o serviço estiver indisponível.

### 4. Testes

- Testar Bot API com fake local; testes não dependem da rede real.
- Cobrir remetente autorizado e não autorizado.
- Cobrir replay de callback, callback de outro run e hash divergente.
- Cobrir falha/timeout da API sem aprovação implícita.
- Cobrir transições `APPROVED -> AWAITING_HUMAN_APPROVAL -> HUMAN_APPROVED`.
- Cobrir notificação de `BLOCKED` sem botão de aprovação.
- Manter `bash -n`, `git diff --check` e os testes existentes do orquestrador
  aprovados.

## Exclusões obrigatórias

- Nenhum terminal, comando shell ou mensagem de texto livre executável.
- Nenhum `/run`, `/stop`, `/cancel`, `/logs` ou controle genérico do host.
- Nenhum commit, cherry-pick, merge, push, deploy ou atualização automática do
  `ROADMAP.md`.
- Nenhum início automático da próxima task.
- Nenhuma integração com OpenAI API nem tentativa de continuar uma conversa do
  Codex pelo Telegram.
- Nenhuma alteração no runtime, API, banco ou containers do produto SaaS.
- Nenhum framework web ou webhook público.

## Critérios de aceite

1. Um `APPROVED` técnico nunca é apresentado como aprovação humana.
2. Somente o `user_id` e `chat_id` permitidos conseguem aprovar.
3. A aprovação corresponde exatamente ao diff revisado.
4. Replays e callbacks adulterados não alteram o estado.
5. Indisponibilidade do Telegram mantém a task pendente, nunca aprovada.
6. `BLOCKED` apenas notifica e não oferece caminho de bypass.
7. O bot não possui interface capaz de executar comandos arbitrários.
8. O worktree permanece para inspeção e integração do planner.
9. Nenhum segredo aparece no Git ou nos relatórios.

## Entrega esperada do executor

- lista exata de arquivos alterados;
- descrição do protocolo de estados e do vínculo run/diff/aprovação;
- comandos e resultados exatos dos testes;
- instruções de configuração sem valores secretos;
- riscos residuais;
- nenhum commit, push, merge ou deploy.
