# DX-08 — Persistência segura, durável e migrável

Status: planejada.

Marco: M1, parte 2 de 2.
Depende de: [DX-07](DX-07.md).
Próxima task: [DX-09](DX-09.md).

## Contexto

Os artefatos mais novos possuem validações no-follow, mas `run.json`, request,
decision, manifesto, delivery e outbox ainda usam leituras comuns em vários
pontos. Escritas fazem `fsync` do arquivo, porém não do diretório após rename ou
hard link. Diretórios e temporários também dependem do umask do operador.

O profile congelado é comparado como dicionário completo e o run não registra a
versão do runner. Mudanças futuras de defaults/schema podem tornar runs antigos
incompatíveis sem uma migration explícita.

## Objetivo

Fornecer uma camada única de persistência que seja:

- segura contra symlink, tipo, owner e modo inesperados;
- durável após crash/reboot dentro das garantias do filesystem;
- transacional para estado e artefatos correlatos;
- versionada e migrável;
- auditável sem exagerar proteção contra processos do mesmo UID.

Ao final desta task, o marco M1 estará concluído.

## Escopo obrigatório

### 1. ADR do backend

Antes de implementar, registrar uma decisão entre:

- JSON + journal/transaction marker e recovery; ou
- SQLite em WAL, com export JSON auditável.

A decisão deve comparar crash consistency, locks multiprocess, migrations,
inspeção manual, portabilidade, backup e complexidade. O backend escolhido deve
cumprir os critérios desta task; não escolher SQLite apenas por conveniência.

### 2. API única de arquivos/estado

Centralizar:

- abertura com `O_NOFOLLOW`;
- `lstat` + `fstat` e verificação de inode;
- arquivo regular, owner esperado e modo sem acesso a grupo/outros;
- tamanho máximo antes e durante a leitura;
- JSON object com schema fechado;
- canonicalização consistente para hashes;
- containment no run/project state canônico.

Aplicar a run metadata, status, failure, heartbeat, snapshots, approval,
rejection, job, delivery, manifesto, outbox, evidence e iteration budget.

Symlink, FIFO, socket, device, hard link inesperado, troca de inode e path escape
devem falhar fechados.

### 3. Permissões e durabilidade

- definir `umask 077` em todos os entrypoints;
- criar state/project/run dirs com `0700`;
- criar locks, temporários e artefatos com `0600`;
- aplicar modo antes de publicar pathname final;
- `fsync` de arquivo e diretório após replace/link/unlink relevante;
- limpar temporários órfãos apenas após validar nome, owner e containment;
- documentar filesystems suportados e limites em NFS/FUSE/network filesystems.

### 4. Transações e recovery

Estado e artefatos exigidos por um evento DX-07 devem ser confirmados como uma
transação lógica. Exemplos:

- `BLOCKED` + `failure`;
- `AWAITING_HUMAN_APPROVAL` + request/outbox;
- `HUMAN_APPROVED` + decision/job;
- `DELIVERING` + delivery progress;
- `PUSHED` + remote OID/delivery final.

Depois de queda em qualquer fronteira, recovery deve concluir a mesma transação
idempotentemente ou restaurar o último estado válido, nunca inventar sucesso.

### 5. Audit trail

Registrar sequência monotônica de eventos com:

- número/ID;
- run id;
- evento e estados anterior/novo;
- hash do evento anterior;
- hashes dos artefatos vinculados;
- timestamp incluído no hash;
- versão do schema/runner;
- origem (`runner`, `bridge`, `worker`, `resume`).

O audit trail não deve guardar tokens, ambiente, URLs com credenciais ou conteúdo
completo de reports.

### 6. Integridade do iteration budget

- incluir `authorized_at`, `updated_at` e hash da entrada anterior no binding;
- validar cadeia inteira, run, feedback e snapshot;
- preservar idempotência de extensões já autorizadas;
- migrar ledger DX-04 legado sem somar orçamento;
- corrupção ou schema futuro deve impedir resume sem reescrever o arquivo.

### 7. Schema e migrations

Adicionar:

- versão do runner e schemas em novos runs;
- registry de migrations explícitas;
- migration dry-run e backup antes de alteração;
- política de compatibilidade com pelo menos duas versões persistidas anteriores;
- recusa segura de versão futura desconhecida;
- testes para runs anteriores a DX-03, DX-04 e ao worker DX-05.

Migration deve ser idempotente e nunca depender de editar JSON manualmente.

### 8. Modelo de ameaça

Documentar dois níveis sem prometer o que não existe:

- baseline suportado: state root `0700`, mesmo UID é confiável; locks/hashes
  detectam corrupção e drift;
- hardened opcional: só pode ser anunciado se chave/autoridade ficar inacessível
  ao executor e testes provarem que ele não forja eventos.

Não implementar HMAC com chave legível pelo mesmo UID e chamá-lo de proteção.
Administrador/root permanece fora do modelo.

### 9. Backup e inspeção

Fornecer comandos read-only equivalentes a:

```bash
agent-loop inspect --run-dir <run>
agent-loop migrate --run-dir <run> --dry-run
agent-loop verify-state --run-dir <run>
```

Saída deve apresentar schemas, hashes, eventos e problemas sem segredos. Backup
deve preservar permissões e permitir restore validado.

## Invariantes de segurança

- leitura nunca segue symlink;
- arquivo inseguro não é “corrigido” automaticamente antes de auditoria;
- migration não promove estado;
- recovery não cria aprovação, decisão ou remote OID;
- timestamp e ordem fazem parte da cadeia;
- versão futura falha sem mutação;
- mesmo UID é declarado como trusted no baseline;
- nenhum segredo entra no audit trail ou backup diagnóstico.

## Compatibilidade e rollback

- migrations criam backup/manifesto antes de escrever;
- rollback restaura somente quando nenhum evento novo ocorreu;
- runs não migráveis permanecem inspecionáveis;
- export JSON continua disponível se o backend for SQLite;
- comandos atuais continuam aceitos ou recebem mensagem de migration necessária.

## Fora de escopo

- proteção contra root/administrador;
- armazenar tokens/segredos no banco ou audit trail;
- cotas/cgroups, tratados na DX-09;
- limpeza/retention, tratadas na DX-10.

## Critérios de aceite

1. Toda leitura/escrita persistente usa uma API segura comum.
2. State/run dirs e artefatos têm modos determinísticos.
3. Rename/link são duráveis com `fsync` de diretório.
4. Cada evento DX-07 é recuperável em qualquer crash point.
5. Ledger inclui timestamps e cadeia anterior.
6. Runs suportados migram idempotentemente com backup.
7. Schema futuro e arquivos inseguros falham sem mutação.
8. Threat model baseline/hardened é honesto e testável.

## Testes obrigatórios

- symlink/hard link/FIFO/socket/device e troca de inode;
- owner/mode incorretos e state root compartilhado;
- kill/reboot simulado antes/depois de write, fsync, replace e dir fsync;
- transações parciais para todos os eventos críticos;
- cadeia de audit/ledger, timestamp adulterado e replay;
- migrations de fixtures DX-01 a DX-07;
- migration repetida, rollback e schema futuro;
- dois processos lendo/migrando/transicionando;
- filesystem sem suporte esperado deve falhar com diagnóstico;
- suíte completa, `git diff --check` e documentação de recovery.

## Entrega obrigatória

- ADR do backend;
- camada de persistência e schemas;
- registry/fixtures de migrations;
- audit trail e comandos de inspeção;
- runbook de backup/recovery;
- matriz de crash points e evidência dos testes;
- modelo de ameaça atualizado.
