# DX-10 — Segredos por fase, streaming e retenção segura

Status: planejada.

Marco: M2, parte 2 de 2.
Depende de: [DX-09](DX-09.md).
Próxima etapa: M3 do [roadmap](../../ROADMAP.md).

## Contexto

`environment.required` é enviado igualmente a bootstrap, Cursor e validações.
Isso pode expor ao agente uma credencial necessária apenas para testes. Diffs,
reports e blobs também possuem caminhos que acumulam todo o conteúdo em disco ou
memória antes de sanitizar/processar.

Não há política operacional de retenção: runs, worktrees, logs e temporários se
acumulam até limpeza manual.

## Objetivo

Concluir M2 aplicando privilégio mínimo por fase, processamento streaming e
limpeza explícita/recuperável, sem alterar os gates de revisão e aprovação.

## Escopo obrigatório

### 1. Ambiente separado por fase

Evoluir o profile para allowlists distintas, por exemplo:

```toml
[environment]
bootstrap_required = ["PACKAGE_INDEX_TOKEN"]
executor_required = []
validation_required = ["TEST_DATABASE_URL"]
```

Requisitos:

- reviewer e delivery nunca recebem ambiente de projeto;
- bootstrap recebe somente sua lista;
- Cursor recebe somente `executor_required`;
- validações recebem somente `validation_required`;
- nomes e valores efetivos são congelados por nome/estado, nunca por valor;
- diagnóstico mostra apenas `NOME=set|unset`;
- env file continua externo, regular, do usuário e `0600`;
- variável extra no arquivo é ignorada;
- profile não pode referenciar `AGENT_TELEGRAM_*`, `GIT_*` perigosas ou nomes
  reservados pelo runner.

O formato legado `required` deve ter migration/depreciação clara. Misturar formato
legado e novo deve ser recusado para evitar interpretação ambígua.

### 2. Credenciais efêmeras

Documentar e suportar arquivos de ambiente gerados antes do run e removidos/rotacionados
depois, sem copiar valores para state ou worktree.

O runner deve:

- abrir o arquivo uma única vez com validações DX-08;
- manter valores apenas na memória necessária para a fase;
- construir novo ambiente a cada subprocesso;
- limpar referências depois da fase;
- não persistir hash reversível, comprimento ou prefixo do segredo;
- sanitizar logs/artifacts com redaction que cubra limites entre chunks.

Não criar cofre próprio nem armazenar credenciais.

### 3. Streaming de subprocessos

Substituir arquivos brutos ilimitados por pipeline incremental:

- contar e limitar bytes conforme DX-09;
- sanitizar antes de publicar logs finais;
- tratar segredo/URL dividido entre chunks;
- manter buffer de sobreposição limitado;
- nunca carregar stdout/stderr completos em memória;
- reports estruturados devem ter limite e validação antes de parse;
- crash não pode publicar conteúdo bruto como log final.

Quando um temporário bruto for inevitável, deve ser `0600`, limitado, nomeado de
forma segura e removido/reconciliado no próximo start.

### 4. Streaming de snapshot e Git

Alterar caminhos que acumulam conteúdo para:

- alimentar SHA-256 incrementalmente;
- processar `git diff --binary`/numstat por pipe;
- ler untracked e manifest entries por chunks com `O_NOFOLLOW`;
- enviar blob a `git hash-object --stdin` por streaming;
- verificar inode, tamanho e mtime antes/depois;
- limitar arquivo/snapshot antes e durante a leitura;
- evitar `b"".join(chunks)` ou `check_output` para conteúdo potencialmente grande.

Resultado/hash/tree devem permanecer byte a byte compatíveis para snapshots
dentro do limite.

### 5. Política de retenção

Adicionar inspeção e limpeza segura, com dry-run default:

```bash
agent-loop cleanup --project-state <dir> --dry-run
agent-loop cleanup --run-dir <dir> --confirm <run-id>
```

Requisitos:

- nunca remover automaticamente run ativo, aprovado não entregue ou worktree com
  mudanças não confirmadas;
- listar exatamente arquivos/worktrees/bytes que seriam removidos;
- exigir confirmação vinculada ao run para ação destrutiva;
- preservar audit trail, delivery final e manifesto conforme política;
- permitir arquivar antes de remover;
- validar containment, owner, symlink e Git common dir;
- nunca aceitar `/`, home, workspace root, glob ou variável vazia como alvo;
- interrupção durante cleanup deve ser recuperável e não deixar outro run parcial.

Automação por idade pode ser proposta apenas para artefatos temporários
reconstruíveis; worktrees e runs exigem decisão explícita na versão inicial.

### 6. Preflight e `doctor`

Exibir:

- backend/limites M2;
- nomes requeridos por fase e set/unset;
- permissões/owner do env e state root;
- espaço disponível e uso estimado;
- temporários órfãos;
- runs elegíveis para arquivamento;
- ausência de segredos no ambiente do reviewer/worker.

Saída deve ser segura para anexar a issue.

### 7. Documentação e modelo operacional

Atualizar:

- schema e exemplo de profile;
- fluxo de credenciais e rotação;
- limites/tipos de falha;
- runbook de disco cheio e output limit;
- backup, archive e cleanup;
- diferenças entre uso pessoal degradado e gate M2 confiável.

## Invariantes de segurança

- cada fase recebe somente sua allowlist;
- reviewer/worker nunca recebem segredos de projeto;
- valores nunca são persistidos;
- redaction funciona entre chunks;
- conteúdo acima do limite não é carregado integralmente;
- streaming não altera hash/tree do snapshot;
- cleanup é dry-run por padrão e nunca segue symlink;
- nenhum run/worktree relevante é apagado sem confirmação específica;
- falha de cleanup não altera aprovação/delivery.

## Compatibilidade e migration

- `environment.required` legado é migrado para comportamento equivalente
  documentado durante janela de depreciação;
- novos profiles devem usar listas por fase;
- snapshots existentes continuam verificáveis;
- hashes antigos não são recalculados silenciosamente com algoritmo diferente;
- cleanup não toca runs de schema desconhecido.

## Fora de escopo

- secret manager/cofre próprio;
- Telegram outbox/offset, tratados no M3;
- limpeza automática de worktree;
- upload de logs/telemetria;
- merge, PR, deploy ou limpeza remota de branches.

## Critérios de aceite

1. Bootstrap, executor e validação recebem ambientes distintos.
2. Reviewer e worker não recebem segredos de projeto.
3. Logs, reports, diffs e blobs são processados com memória limitada.
4. Hash/tree permanece compatível para o mesmo snapshot.
5. Segredo dividido entre chunks é redigido.
6. Arquivo/snapshot/output grande falha com reason estruturado.
7. Cleanup é seguro, explícito, auditável e recuperável.
8. Todos os critérios M0, M1 e M2 passam em CI.

## Testes obrigatórios

- segredo exclusivo de cada fase e variável reservada;
- formato legado, novo e mistura recusada;
- secret/URL quebrado em todas as posições de chunk;
- stdout/stderr/report no limite e acima;
- diff/binário/untracked/blob grande com memória medida;
- mudança de inode/mtime durante streaming;
- equivalência de hashes/tree antes e depois;
- disco cheio e temporário órfão;
- cleanup dry-run, confirmação errada e run ativo;
- symlink/path escape/home/root e interrupção durante cleanup;
- doctor sem valores sensíveis;
- suíte completa e integração M0–M2.

## Entrega obrigatória

- schema/migration de ambiente;
- pipeline streaming e limites;
- comandos doctor/cleanup/archive necessários;
- documentação de credenciais e retenção;
- testes de memória, redaction e destruição segura;
- relatório final comprovando os gates M0–M2.
